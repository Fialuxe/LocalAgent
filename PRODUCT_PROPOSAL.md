# LocalAgent プロダクト提案書
*10エージェント並列調査による統合レポート — 2026-06-21*

---

## プロダクトビジョン

> **「あなたの作業状態を空気で読み、話しかけるタイミングを知っているローカルAIコンパニオン — 完全オフライン、キャラクターはあなたのもの。」**

Clippyが失敗したのは「タイミングを持たない介入」のせいだった。  
Replikaが批判されたのは「実務なき感情依存」のせいだった。  
LocalAgentはその両失敗の空白を埋める。

## 3. UXアーキテクチャ

### 3-1. 存在形態：ハイブリッド構成（推奨）

```
通常時:  小型デスクトップペット(64px, 画面右下) + トレイアイコン
呼んだとき: ホットキー → フローティングウィンドウ展開 + キャラ拡大
通知時:  OSネイティブ通知 + キャラクターアニメーション
詳細確認: サイドパネル (オプション、右端スライドイン)
```

### 3-2. 5段階の作業フェーズ認識

| フェーズ | 検出条件 | AIの振る舞い |
|---------|---------|-------------|
| Deep Focus | 同一ウィンドウ20分以上、規則的入力 | **完全サイレント**。通知はキューに積む |
| Active Work | 複数ウィンドウ切り替え、通常作業 | Level2以上の通知のみ。プル型はいつでも |
| Scattered | 断続的入力、頻繁なウィンドウ切り替え | 優先タスク確認を軽く提案 |
| Break | 在席+アイドル5分以上 | 会話的プッシュを許可。天気・予定を提案 |
| Away | 顔検出なし | バックグラウンドタスク実行。帰還時サマリー |

### 3-3. 割り込みの設計原則

- **最後のキーストロークから2分以内は一切の割り込み禁止**
- 通知は画面の端（右下）のみ。作業中心（画面中央）を侵犯しない
- 同カテゴリの通知は30分クールダウン
- Deep Focus中の蓄積通知はフェーズ終了後に「3件の情報があります」でまとめ提示

### 3-4. Clippy失敗から学ぶ設計の三原則

1. **タイミング**: 気づいているときに反応する、介入は遅延評価する
2. **文脈感度**: ユーザーが何をしているかを理解してから行動する
3. **可逆性**: すべての通知はスヌーズ・延期・ミュート可能

## 4. キャラクター設計

### 4-1. アーキタイプ：「守護精霊 × コンパニオン」

ユーザーより小さく弱い存在が、ユーザーのことを最もよく知っている——という逆説的な関係性。Clippyは「教師」だったから嫌われた。このキャラクターは「相棒」である。

### 4-2. パーソナリティ（Big Five）

| 特性 | スコア | 理由 |
|------|-------|------|
| 開放性 | 75 | 情報への好奇心、新しいことへの反応 |
| 誠実性 | 70 | 実務的信頼感（完璧主義にはしない） |
| 外向性 | 65 | 積極的だが押しつけがましくない |
| 協調性 | 80 | 共感・受容（90超はイエスマンになる） |
| 神経症傾向 | 30 | 安定感を保つ（時々「少し心配」は残す） |

### 4-3. 口調設計：「柔らか敬語」

**コアの口癖:**
- `むむ？` — 何かを検知したときの反応（断言でなく疑問形）
- `えーと、` — 考える間を作る
- `いっしょに〜しましょうか` — コンパニオン性の言語表現
- `〜って、知ってましたか？` — 情報共有を発見の共有にする
- `よかった` — 短く純粋な喜び

**口調サンプル:**
```
朝: 「おはようございます！ …むむ？ 今日はちょっと眠そうですね。
     あのですね、今日の天気、雨らしいですよ。傘、持っていくと安心かもです。」

集中中: 「……（静かに）集中してますね。邪魔しませんよ。
        でも、肩、力が入ってますよ。少し伸ばしてみてください。それだけ。」

タスク完了: 「わあ、終わりましたね！ よかった。……えへ。
           見てましたよ、ずっと。お疲れさまでした。」

深夜: 「……むむ。遅いですよ。続きは、明日の朝、わたしが
      一緒に覚えてますから。……今日は、もう休んでください。」
```

### 4-4. 関係性ティア（信頼の成長）

```
Tier 0 出会い    → 基本挨拶のみ、少しシャイ
Tier 1 顔見知り  → 天気・カレンダー連携開始（3日連続起動で解除）
Tier 2 友人     → 感情表現が豊かに（通算10時間共存で解除）
Tier 3 相棒     → ユーザーの習慣を把握した言動（30日アクティブで解除）
Tier 4 親友     → 過去会話を参照した深い会話（特定イベントで解除）
```

**重要設計:** ティアは**下がらない**。長期離脱後は「寂しかった」演出で再接続を促すのみ。罪悪感でなく愛着で継続を促す。

### 4-5. キャラクタービジュアル（段階的実装）

| フェーズ | ビジュアル | 技術 |
|---------|-----------|------|
| MVP | ピクセルアート（64x64px） | PixiJS スプライトシート |
| Phase 2 | Live2D 2Dアニメキャラ | Cubism SDK for Web + Electron |
| Phase 3 | VRM 3Dモデル対応（オプション） | Three.js または Unity |

**感情5ステート（MVP最低限）:**
待機 / 喜び / 集中 / 眠い / 驚き

## 5. ゲームデザインメカニクス（エンゲージメント設計）

### デイリーループ
```
起動 → キャラが目覚める演出 → 当日一言（天気・予定）
→ 共存フェーズ（作業認識・適切な介入）
→ 日締め（今日の振り返り・翌日の予告）→ スリープ演出
```

### 「かわいい」インタラクション演出トップ5

1. **眠り込みモーション** — 15分離席でうとうと、帰宅で慌てて起きる
2. **一緒に伸びをする** — ポモドーロ終了時に「一緒に伸ばしてね」
3. **頭を傾けるキョトン反応** — 同じ画面で止まっているとき小首を傾げる
4. **雨の日の特別演出** — 天気APIと連動し傘を持って登場
5. **記念日サプライズ** — 100日目・365日目に小さなパーティー演出

### エンゲージメント原則

- **可変報酬**: 「内緒話」モーション（週1〜2回ランダム）で稀少感
- **損失ペナルティなし**: 離脱してもキャラは消えない、ティアは下がらない
- **相互成長**: キャラが「ユーザーから学んだ」と言語化する

## 6. セキュリティ・プライバシー設計

### 最重要: カメラプロセス分離

```python
# カメラ処理プロセス（ネットワーク無効）
カメラ映像 → MediaPipe推論 → テキストラベル("present"/"away")
                ↓ IPC (名前付きパイプ) のみ
           メインプロセス（ネットワークあり）
```

カメラフレームはメモリ上で処理後即時破棄。ディスク・ネットワークへの書き出しは絶対禁止。

### P0 実装チェックリスト（リリース前必須）

- [ ] カメラ処理プロセスとネットワーク処理プロセスを分離
- [ ] 会話履歴はAES-256-GCM + Windows DPAPIで暗号化
- [ ] APIキーはソースコードになし（`keyring`でCredential Manager管理）
- [ ] 全外部API通信はHTTPS必須、証明書検証を無効化しない
- [ ] OAuth認証にPKCE + stateパラメータを実装
- [ ] システムトレイにカメラ使用状態インジケーター常時表示
- [ ] カメラ機能はデフォルトOFF、明示的オプトイン

## 7. アクセシビリティ設計

### 認知負荷最小化の7原則

1. デフォルト状態はシステムトレイのみ（ゼロ注意状態）
2. 注意の段階的要求（5段階: 不可視→微細アニメ→ソフト音→バナー→割り込み）
3. 最後のキーストロークから2分以内は割り込み禁止
4. アクティブウィンドウからコンテキスト推測（会議中=完全サイレント）
5. 通知の消費期限管理（後読みキューに積む）
6. 明示的召喚（ホットキー）を最優先フロー
7. すべての通知はスヌーズ・ミュート可能

### ADHD・ASD配慮

- アイドルアニメーションはデフォルトOFF（動くものが気になる）
- `prefers-reduced-motion` OS設定への追従
- シングルタスクモード（今やることを1つだけ表示）
- スケジュールの事前提示（「今日は10時と15時にお声がけします」）
- フォントはBIZ UDゴシック（ユニバーサルデザイン）

### 5段階「邪魔度スライダー」

```
レベル1（超静穏）: 手動召喚のみ
レベル2（低干渉）: 緊急のみ、1日最大3回
レベル3（標準） : フロー保護あり、1時間最大2回
レベル4（能動的）: 積極的提案あり
レベル5（おしゃべり）: 最大頻度
```

## 10. 今すぐ始めるためのステップ（最初の72時間）

```bash
# Step 1: Ollama インストール
winget install Ollama.Ollama

# Step 2: モデル取得
ollama pull qwen2.5:7b

# Step 3: VOICEVOX をダウンロード・起動
# https://voicevox.hiroshiba.jp/ からインストーラーを取得

# Step 4: 動作確認スクリプト（Python）
pip install ollama requests

# Python で以下を実行:
import ollama, requests, json, subprocess

response = ollama.chat(
    model='qwen2.5:7b',
    messages=[{'role': 'user', 'content': 'こんにちは！今日もよろしくね。'}]
)
text = response['message']['content']

# VOICEVOX で音声合成
query = requests.post(
    'http://localhost:50021/audio_query',
    params={'text': text, 'speaker': 3}
).json()

audio = requests.post(
    'http://localhost:50021/synthesis',
    params={'speaker': 3},
    data=json.dumps(query)
).content

with open('response.wav', 'wb') as f:
    f.write(audio)

subprocess.run(['start', 'response.wav'], shell=True)
```

このループが動いた瞬間、プロダクトのコアバリューが体感できる。
